# IDE-first 운영 모델

목표는 사람들이 원고 검수/교정을 웹 페이지 업로드 작업으로 끝내지 않고, IDE 안에서 반복 가능한 작업으로 다루게 만드는 것입니다.

## 왜 IDE인가

원고 검수는 한 번의 질문/답변으로 끝나지 않습니다.

- 작품이 여러 개다.
- 회차가 계속 추가된다.
- 같은 문제 패턴이 다른 작품에도 반복된다.
- 리포트, 교정안, export, 승인 이력이 남아야 한다.
- AI가 한 번 놓친 문제를 다음 run에서 다시 놓치지 않게 해야 한다.

채팅창만 쓰면 이력이 흩어집니다. IDE는 원고, 리포트, 스키마, 템플릿, run manifest를 한 화면에 두고 모델과 같이 고칠 수 있습니다.

## 사용자 경험 목표

사용자는 repo를 열고 다음 세 가지를 반복하면 됩니다.

1. 작품 등록
2. run 생성
3. 에이전트에게 run 폴더를 기준으로 검수/교정/보고서 생성을 맡김

```powershell
.\scripts\novel-qc-loop.ps1 list-works
.\scripts\novel-qc-loop.ps1 start-run --work canaria --kind adversarial-audit
.\scripts\novel-qc-loop.ps1 portfolio-status
```

## IDE 안에서의 역할 분담

사람:

- 작품 의도, 플랫폼, 대상 독자, 작가 성향을 정한다.
- `ⓐⓐ` 작가 판단 항목을 승인하거나 반려한다.
- 최종 보고서의 말투와 전달 수위를 결정한다.

AI/에이전트:

- 긴 원고를 나누어 읽고 문제를 구조화한다.
- 반복 표현, 모바일 호흡, 수치/시간/장소 문제를 찾는다.
- 내부 감리 결과를 human-facing 보고서로 변환한다.
- 승인된 교정안을 HWPX/PDF/MD export로 정리한다.

Repo:

- 작품별 작업 공간을 분리한다.
- run 단위 증거와 리포트를 남긴다.
- 다른 사람이 같은 규칙으로 다시 실행할 수 있게 한다.

## 반복 루프

```text
작품 등록
  -> 원고 투입/추출
  -> 1차 전역 감리
  -> 적대적 감리 3회
  -> 교정안 생성
  -> human-facing 보고서
  -> export
  -> 다음 회차/다음 작품으로 반복
```

## 에이전트 프롬프트 예시

```text
workspace/canaria/runs/{run_id} 기준으로 작업해.
source_text_path의 원고를 읽고 적대적 감리 3회 수행해.
P0/P1/P2를 분리하고, 작가-facing 보고서는 final_reports에 작성해.
원고 원본은 절대 덮어쓰지 마.
```

```text
portfolio-status를 보고 pending run이 있는 작품부터 처리해.
각 작품의 manifest를 읽고 플랫폼/대상 독자 기준에 맞춰 검수 수위를 조정해.
```

## 성공 기준

- 새 작품을 넣을 때 폴더 구조를 다시 고민하지 않는다.
- 리포트 파일 이름과 위치가 매번 같다.
- 여러 작품의 pending 상태가 한 명령으로 보인다.
- 어떤 AI 모델을 쓰든 작업 산출물이 같은 manifest/schema에 맞춰 남는다.
- 팀원이 IDE로 repo를 열면 현재 어디까지 왔는지 이해할 수 있다.
